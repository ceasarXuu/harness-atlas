# 完整开发现场备份与恢复操作说明

## 目的

在设备重置、迁移或故障前，不能只依赖远端 Git。完整开发现场还包括 Git 本地元数据、忽略目录、代理会话、自动化、项目技能、编辑器状态、本机配置和活跃数据库。

## 备份分层

1. **项目仓库**：完整复制工作区，必须包含 `.git` 和所有隐藏/忽略文件。
2. **项目代理现场**：只保留能够通过项目路径、自动化名称或项目技能名称证明归属当前项目的 Codex/Claude 会话、自动化与项目技能。
3. **环境清单**：记录操作系统、CPU 架构、Node/npm/Git/gh/Codex、Homebrew 包、全局 npm 包和编辑器扩展版本，但不复制机器级凭据和其他项目数据。
4. **校验**：对最终文件树生成 SHA-256 清单，并执行 Git、测试与构建验证。

## 容易出错的地方

- 全量复制 Codex/Claude 状态库会混入其他项目数据。项目专项备份应优先筛选项目路径、自动化目录、项目技能和项目会话；需要整机迁移时再使用独立的整机备份方案。
- Git 工作树干净不代表 `.git` 可丢弃。reflog、被改写提交、stash、自定义 hook 和本地 refs 都只存在于本机；完整现场备份应保留整个 `.git`。
- `node_modules`、`.astro` 和 `dist` 理论上可重建，但离线恢复、依赖下架或工具链变化时可能无法字节级复现；完整现场模式仍应复制。
- Unix socket、进程锁和 IPC 文件不是可恢复数据。记录路径即可，恢复后由对应应用重建。
- 复制成功不等于备份可用。至少需要文件校验、数据库完整性检查、Git 检查和一次临时恢复演练。

## 推荐目录结构

```text
harness-atlas-完整开发现场-YYYY-MM-DD/
  00-恢复文档/
  01-项目仓库/harness-atlas/
  02-项目关联现场/
  03-环境清单/
  04-校验结果/
```

## 恢复顺序

1. 核对备份根目录校验清单。
2. 安装清单中记录的基础开发工具。
3. 恢复项目仓库和项目关联代理现场，保留权限、时间戳、符号链接和扩展属性。
4. 检查并修正自动化和配置中的旧绝对路径。
5. 执行 `git fsck --full`、`git status`、项目测试和构建。
6. 机器级 GitHub/Codex/Claude 登录及推送权限由使用者单独恢复。

## 验证记录要求

每次备份都应记录：来源路径、目标路径、文件数量、逻辑大小、SHA-256 清单摘要、SQLite 检查结果、Git HEAD/远端状态、测试结果、构建结果和演练时间。
